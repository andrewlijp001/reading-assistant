# 読書ツール — 桌面应用 (Electron) 改造纪实

本文档详细记录了本项目从一个依赖于终端运行的纯网页版（Web App）升格为具备原生视窗体验的桌面应用（Desktop App）的开发前因后果及技术演进策略。

---

## 💡 一、前因后果 (Why Desktop App?)

### 网页版的痛点
1. **启动繁琐**：每次使用前，必须打开 Terminal，进入项目目录，运行 `npm run dev`，然后再单独打开浏览器输入网址。不够“随叫随到”。
2. **沉浸感不足**：浏览器的标签页、地址栏以及其他网页会极大干扰读电子书时的心流体验。
3. **数据交错风险**：网页版的知识沉淀（如生成的 `.md` 笔记）与代码本身混杂在同一物理目录 (`history/`) 之中。如果不熟悉 Git 操作，用户极其容易在清理代码时误删辛勤积攒的笔记。

### 改造核心诉求
- 我们需要一个像 `VS Code` 或 `Notion` 一样，双击就能启动在底层任务栏（Dock）的**真正原生软件**。
- 但是，**绝对不能舍弃网页版的代码**，要让两端架构 100% 一致，维护一套代码即可双端生效。

---

## 🏗️ 二、核心方案设计：双轨融合架构

为了在这个 React + Node.js(Express) 技术栈里植入原生生命，我们引入了工业标准打包容器 **Electron**，并精心设计了以下策略：

### 1. 隐蔽后台挂载（无需改写接口）
传统的 Electron 项目往往需要将通信系统全部按 IPC 协议（`ipcMain` 与 `ipcRenderer`）进行重构，代价极大。
我们采用了极客的**“包裹策略”**：在 Electron 的核心大脑文件 `main.js` 启动原生存边框透明窗口的同时，立刻执行 `import('./server.js')`。这样，Electron 自己会在底层“偷偷开一个本地服务器”，以前负责翻译、写笔记的逻辑连一寸代码都无需改动！

### 2. 魔法路径迁移：应对「沙盒封锁」
打包成 `.app` 之后的终极难关在于系统环境差异。Mac 环境对 `app.asar` (打包后的软件主图) 做了绝对的**只读加密保护**。之前的应用一直都是在 `__dirname` 本地根目录下创建和改写 `history/` 笔记文件的，一旦打包变成软件就会全盘崩溃失效。

为了打破僵局，我们开发了**环境变量侦查通道**：
在 `main.js` （原生层）里使用原生 API 查询 Mac 系统的专属隐私位置：
`process.env.STORAGE_PATH = app.getPath('userData')`
并且修改了后端的路由，只在这个变量存在时启动异源写入。这样，不仅突破了只读沙盒，还极其自然地实现了用户数据（私人笔记）和代码运行环境的**物理性隔离防护**。

### 3. 热切换编译基座
对于前端显示逻辑，我们巧妙利用了 `cross-env` 设置变量以判断现在是调试过程还是发版过程：
- 开发时：`main.js` 加载 `http://localhost:5173`。随时改代码、原生窗口中随时热刷新。
- 收包时：通过 Vite 追加了相对路径保护 `base: './'` 的编译结果。`main.js` 转为直接读取硬盘静态文件加载。

---

## 🎯 三、总结：Maintainability 跃迁
这套方案让未来的一切二次开发都充满了弹性 —— 如果在此后增加比如『语音合成连读』等新功能，直接在 `src/(即 React 网页池)` 里更新逻辑即可，不仅下次打开 App 自动带有发音键，连传统网页版依然还能丝滑打开同步使用。这，就是“一次编写，处处融合”的极高开发心智模型！
