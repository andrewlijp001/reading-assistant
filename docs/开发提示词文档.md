# 読書ツール — 开发提示词文档

本文档记录了「読書ツール」从需求沟通、Claude API 初版建立，到最终进化为搭载 Google Gemini 2.5 Flash 模型、支持双模态面板及防抖沉淀系统的现代化工具的完整开发历程。

---

## 一、需求沟通阶段（初版回忆录）

**业务初衷**
- 主要读 PDF 和 EPUB 电子书，英文约 CET6 水平，日文约 N2 水平。
- 网页版，左读右析，支持查词时显示中文释义 + 例句，日语全假名。

## 二、Phase 2 进阶突破与痛点优化 (重构阶段)

在初版架构落成后，根据日常使用体验遭遇的壁垒进行了深度重构与 API 迁移。

### 2.1 核心驱动引擎更换
> **痛点**：Anthropic 平台网络连通性或费用计价限制了超长文本处理（全书分析）。
**重构动作**：
将服务端引擎全面从 `@anthropic-ai/sdk` 迁移至最新发布的 `@google/genai` 官方 SDK。使用性价比最高且响应极其迅捷的 `gemini-2.5-flash` 模型处理全书数万到数十万字的总结。环境变量由 `ANTHROPIC_API_KEY` 更换为 `GEMINI_API_KEY`。

### 2.2 多语言强制输出自适应
> **提示词进化**：原本粗放式的指令被升级为“分歧提示词系统”。
实现逻辑：在调用 Gemini 接口前，通过正则 `[\u3040-\u309F\u30A0-\u30FF\u4E00-\u9FAF]` 检测用户投喂的本文语言。如果检测到日语，强制将 System Prompt 封装为日语系，专注提取 N3+ 难度语法；如果检测到英语，则封装为浸入式英英解析界面并附带 CET-4+ 高级语法筛查。

### 2.3 工作流演进：防呆拦截与多功能提示词
> **交互问题**：用户在使用“全书分析”或“AI 自由分析”时，想要临时改变询问主题，却苦于没有特定的参数传递入口。其次，EPUB 解压大书时内容滞后，使用户经常误点瘫痪组件。
**修复方案**：
引入了公用的「✨ 自定义提示词框」。打通了从前端 `React state` 到后端 `req.body.prompt` 的参数透传链路。不论有无选中文本，一旦用户敲入自定义字符，按钮立即解冻高亮（点亮规则引擎覆盖 `isDisabled` 钩子），并且加入了安全回退的错误拦截提示。

### 2.4 全新数据落石策略：隐式本地归档与多轨 Tab
> **设计决断**：学习工具最大的问题是“学过即忘”，单纯的前台 State Array 难以沉淀价值。用户希望能够把查词历史与随笔感受彻底沉淀。
**技术实现**：
1. 后端依赖 Node `fs` / `path` 模块，新增核心守护进程函数 `saveHistory`。将全站所有的 AI 响应统一无感并行持久化写入 `history/[bookName]/[功能名].md` 文件中。
2. 为统一语境美感且无惧跨操作系统乱码，所有实体落地 MD 文件名在最后一轮升级中全盘英文化重构（`Vocabulary.md`, `Grammar.md`, `Book_Analysis.md`, `AI_Analysis.md`, `Notes.md`）。
3. 前端界面使用 Tailwind 构筑深色玻璃态 Tab 导航器，借助 absolute 和 opacity 的淡入淡出堆叠，实现了 `AIPanel` 和全新创建的 `NotesPanel` 两大组件互切不丢状态（防组件卸载）的高级操作。
4. `NotesPanel` 中引入了经典的 `setTimeout` 防抖 (Debouce) 心跳锁逻辑，使得用户可以流畅高频书写 Markdown，一旦停顿便在后台触发全异步的 `/api/notes` POST 落地。

### 2.5 EPUB 健壮性剥离
> **隐秘危机**：少数特殊 EPUB (比如图文集/强加密书) 在内部解析其 Spine 数据链时 `item.document` 会发生内存访问崩溃，致使全书提取永远不会 throw 异常，也不回调，让 UI 永久陷入阻塞不可点状态。
**硬核加固**：重写了 `EPUBReader.jsx` 内核。采用递归 try-catch 和多层级安全 DOM 读取（优先 `body.textContent`，次之 `textContent`，最差用正则 `replace(/<[^>]*>?/g)` 剥离 html 壳），最后提供保底“兜底填充值”抛给上级，百分之百保证状态机必定成功闭环。
